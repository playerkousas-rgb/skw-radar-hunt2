# Radar Hunt 架構說明

## 資料流動與限制

### 目前架構（純前端 PWA）

```
領袖手機
  ├── 創建地圖（寶藏點含經緯度、內容、觸發半徑）
  ├── 匯出為 JSON / QR Code / 連結
  └── 分享給成員

成員手機
  ├── 匯入地圖（獲得完整寶藏資訊）
  ├── 使用 GPS 尋寶（必須真實移動到位置）
  └── 僅僅觸發時記錄到自己手機

限制：
❌ 沒有中央伺服器
❌ 成員之間無法同步進度
❌ 領袖無法即時看到成員進度
❌ 排行榜只是本地存於該手機
```

## 你提出的關鍵問題

### 1. **成員知道所有答案**
因為資料存在 JSON/QRCODE 中，成員匯入後就有完整地圖資訊。

**解決方案：**
- 開發模式：提供"模擬尋寶"按鈕讓測試更容易
- 正式遊玢：必須信任成員不會偷看答案，或...
- 後端架構（見下面）：寶藏點坐標存在伺服器，成員只有 ID

### 2. **即時排行榜**怎麼實現？

**目前（本地式）：**
- 只是該手機的歷史記錄
- 無法看到其他玩家的成績

**要真正實現需要：**
- 後端伺服器（Firebase / Supabase / 自訂 API）
- 成員完成後上傳成績
- 領袖可查看所有人進度

## 建議的後端架構

```
推薦方案 A：Firebase 免費方案
- Firestore 存地圖、寶藏點、成員進度
- Realtime Database 即時同步
- Authentication 識別領袖/成員

推薦方案 B：Supabase 免費方案
- PostgreSQL 資料庫
- Realtime subscriptions
- 非常適合排行榜功能

推薦方案 C：自訂後端（主機代管）
- Node.js + Express + MongoDB
- WebSocket 即時通訊
- 完全控制資料
```

## 立即可實現的改進

不需要後端的改進方案：

1. **加密地圖資料**
   - 領袖用密鑰加密寶藏座標
   - 成員只有密文，難以解讀

2. **分段提供資訊**
   - 只提供第一個寶藏位置
   - 找到後才給下一個提示

3. **領袖回報機制**
   - 成員完成後點擊"上報成績"
   - 產生一個成績代碼給領袖
   - 領袖匯入後更新排行榜

4. **用手機當伺服器**
   - 領袖手機開啟 hotspot
   - 成員連到領袖的 WiFi
   - 用 WebRTC 或區域網路通訊

需要後端的功能：
- 即時看到所有人位置（這是隐私問題）
- 即時排行榜更新
- 防止偷看答案
- 多人即時协作

## 建議

如果這是小型活動（<50人）：
- 目前架構已足夠
- 領袖可以用手動收集成員回饋

如果是大型活動或商業應用：
- 必須建立後端
- 推薦 Firebase 或 Supabase 免費方案開始
- 預估開發時間：2-4 週
